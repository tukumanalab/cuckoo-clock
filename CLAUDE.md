# Cuckoo Clock — プロジェクト概要

## プロジェクトの目的

オルゴール（ミュージックボックス）用カムの STL ファイルを生成する Web アプリ。
3D プリントで製作したカムを実際のオルゴール機構にセットして使用する。

## ファイル構成

| ファイル | 内容 |
|---|---|
| `index.html` | UI レイアウト（メロディエディター・3D プレビュー・各種ボタン） |
| `app.js` | STL/SVG 生成・Three.js プレビュー・ピアノロール・MIDI 読み込み/保存 |
| `style.css` | スタイル |

## カム仕様

| パラメーター | 値 |
|---|---|
| セクター数 | 25 等分 |
| 外径パターン | φ40 → 37.5 → 35 → 32.5 → 30 mm × 5 回繰り返し |
| 厚さ | 1.8 mm |
| 中央穴 | 10.5 × 10.5 mm 正方形 |

## 音階解析結果（2026-05-15 時点）

### 前提
- **大きい径 = 高い音**（ユーザー確認済み）
- forward.m4a: φ40 → φ37.5 → φ35 → φ32.5 → φ30 の順（高 → 低）
- backward.m4a: φ30 → φ32.5 → φ35 → φ37.5 → φ40 の順（低 → 高）
- 5音 × 5サイクル = 25音（25セクターと一致）

### 5音サイクルカム解析（forward.m4a / backward.m4a）

| 径 | 推定音名 | 観測周波数 | 備考 |
|---|---|---|---|
| φ40mm（最高音） | **A7** | ~3436Hz | A7(3520Hz)より約42¢低め |
| φ37.5mm | **F#7** | ~2980Hz | F#7(2960Hz)より約12¢高め、ほぼ正確 |
| φ35mm | **E7** | ~2708Hz | E7(2637Hz)とF7(2794Hz)の中間あたり |
| φ32.5mm | 不明 | 不安定 | サイクルごとにスペクトルが変動 |
| φ30mm（最低音） | 不明 | 不安定 | 同上 |

## 単音カム解析結果（2026-05-20 時点）

単音カム（全25セクター同一径）を用いた精密測定。

### φ30mm（low.m4a）

- 打鍵検出数: **28回**
- 主要ピーク: **2392Hz** → **D7 +31¢**（D7標準 = 2349Hz）
- 二次読み: 2449Hz ≈ D#7 -28¢
- **結論: D7 / D#7 付近（約2400〜2450Hz）**

### φ35mm（mid.m4a）

- 打鍵検出数: **36回**
- 主要クラスター: 3333〜3464Hz、中心 **3406Hz → G#7 +43¢**（G#7標準 = 3322Hz）
- 非常に安定した単峰性スペクトル
- **結論: G#7（Ab7）、約+40¢ シャープ**

### φ40mm（high.m4a）

- 打鍵検出数: **25回**
- G#7クラスター(3296〜3416Hz): φ35mm と共通 → **ボディ共鳴（胴鳴り）**
- φ40mm 固有ピーク（φ35mm との差スペクトルより）:
  - 3758Hz (Bb7 +13¢)、3659Hz (Bb7 -33¢)、930Hz (A#5)、853Hz (G#5)
- **結論: 真の音名は Bb7 / A#7（約3660〜3760Hz）**
  - G#7 ピークは φ35mm と共有するボディ共鳴であり、琴爪の固有音ではない

### 単音カムまとめ

| 径 | 実測音名 | 代表周波数 | 備考 |
|---|---|---|---|
| φ40mm | **Bb7（A#7）** | ~3709Hz | G#7ピークはボディ共鳴、差分で同定 |
| φ35mm | **G#7（Ab7）** | ~3406Hz | 非常に安定 |
| φ30mm | **D7〜D#7** | ~2420Hz | D7よりやや高め |

### 分析上の注意点
- 音の基音が 2500〜4000Hz と高いのは、カムが小径（最大φ40mm）のため琴爪が短く高周波になるため
- 単音カムでは同じ音が25回繰り返されるためボディ共鳴が強く出やすい
- φ40mm と φ35mm が同じ G#7 付近のピークを持つのは共通ボディ共鳴が原因
- 解析ツール: Python + soundfile + scipy（macOS `afconvert` で m4a → WAV 変換）

## 曲のインポート（MIDI 経由）

曲は **MIDI ファイル（SMF）** でアプリと読み書きする（2026-06-04 にドレミ txt から置き換え）。
UI は「MIDIファイル読込」「MIDIファイル保存」ボタン（`loadBtn` + hidden `loadFile` / `saveBtn`）。

### MIDI 読み込み仕様（`app.js` の `parseMidi()` / `midiNoteToIndex()`）
- SMF format 0/1 対応（SMPTE 時間形式は非対応）。全トラックのノートオンを tick 順に統合
- 1 拍 = 4 分音符として `round(tick / division)` 拍目のセクターに割り当て。音のない拍は休符
- 音高はオクターブ畳み込みで `NOTE_SOLFEGE`（`ラ ソ# ソ ファ# ファ ミ レ# レ ド# ド` = A〜C の
  半音階 10 音）へ対応付け。A#・B 系は畳み込めないため無視
- 25 拍超過・音域外・同一拍重複（和音は先勝ち）は無視し、件数を alert で通知
- 最初のテンポメタを試聴テンポ（`bpm`、初期 ♩=300）に反映。試聴と MIDI のテンポは常に一致させる
- 実例: `sound_files/AMA2.midi` / `sound_files/FAN2.midi`

### MIDI 保存仕様（`app.js` の `melodyToMidi()`）
- format 0・1 トラック・480tpq・拍子 25/4・テンポは現在の `bpm`（サンプル MIDI と同じ規約）
- 1 音 = 4 分音符（長さ 360tick・ベロシティ 95）、音高は A5(81)〜C5(72)、休符は無音区間

### 旧フォーマット: ドレミ txt（アプリ読み込みは廃止）
- 1 音 = 1 トークン、空白/改行区切り。休符は `-`、行頭 `//` はコメント
- `sound_files/AMA2.txt` / `FAN2.txt` は PDF 判読結果の記録として残置

### PDF 楽譜 → ドレミ txt の判読ロジック（OMR）

MuseScore 出力 PDF（音符は Bravura グリフ描画＝テキスト化不可）から音高を起こす手順:

1. **画像化**: `pdftoppm -png -r 600 sound_files/AMA2.pdf /tmp/ama2hi`
2. **五線検出**: PIL でグレースケール→2値化（`<128`=黒）。左側（例 x<2200）の行黒画素プロファイルの
   山から五線 5 本の y を取得。線間隔 `d`（AMA2 では 600dpi で約 40px）。段が複数あれば y で帯分割。
3. **五線消去**: 各行で連続黒ランが `>3d` のものを白化（長い五線は消え、符頭・符幹は残る）。
4. **符頭抽出**: 自前のランベース連結成分（scipy は numpy2 系と非互換で使えないため自作）。
   符頭フィルタ = 幅 `0.7d〜2.0d`・高さ `0.6d〜1.4d`・面積 `>0.4d²`。符幹（縦長）や音部/拍子記号は除外。
5. **音名化**: 符頭中心 y を五線基準で判定。最上線 = F5、`d/2` ごとに 1 ダイアトニック段
   （F5,E5,D5,C5,B4,A4,G4,F4,E4,D4,C4…）。
6. **誤検出対策**: 冒頭は音部記号・拍子記号（25/4 の数字）を符頭と誤認しやすい。本体は等間隔
   （AMA2 は 600dpi で約 167px ピッチ）に並ぶので、その格子に乗らない成分を除外し、左端数音は拡大画像で目視確認。
7. **♯の扱い**: ♯は符頭の左の小グリフ。自動では拾わず、♯が付く音を目視で特定して半音上げ。
8. **アプリ音域へ**: 実音をそのまま `NOTE_SOLFEGE` に対応付け。AMA2 は音域 A4〜D4、FAN2 は A4〜C4 で
   いずれも 10 音内に収まり移調不要。範囲外なら最寄り音へ丸めるかオクターブ畳み込み。

#### OMR が取りこぼしやすい音（必ず本数を検算する）

自前 OMR は符頭サイズ・形状フィルタで拾うため、次の音は**検出漏れしやすい**。総数が 25 に合わない
ときは、まず以下を疑い、等間隔格子の欠けた位置を拡大画像で目視補完する:

- **♯付きの音**: ♯記号が符頭にくっつき、連結成分が大きく/変形して幅・高さフィルタを外れる
  （FAN2 の 3 音目 D#4 がこれで欠落）。
- **下加線（ledger line）付きの低音**: 加線が符頭に連結して横幅が広がり、`幅 ≤ 2.0d` を外れる
  （FAN2 の 19 音目 C4 がこれで欠落）。
- **拍子記号に重なる冒頭の音**: timesig の数字を符頭と誤検出する一方、本物の冒頭音が埋もれることがある
  （AMA2 の冒頭 2 音）。

検算の指針: 本体は等間隔ピッチ（600dpi で約 167px）で並ぶので、`(最終x − 先頭x) / ピッチ + 1` が
音数になるはず。OMR の検出数とずれたら、その格子穴を拡大して読む。1 段目＋2 段目の合計が 25
（= セクター数）になることを必ず確認する。
